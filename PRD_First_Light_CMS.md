# PRD: First Light CMS

**Version:** 0.1 (Draft)
**Author:** Dustin Wahlquist
**Date:** 2026-05-24
**Platform:** Web app (internal tool)

---

## 1. Overview

The First Light CMS is an internal web tool for managing all bird species content that powers the First Light mobile app. Every bird card players collect is backed by shared species data — stats, descriptions, facts, art, and moves — that lives in the `bird_species` table and an upcoming `species_moves` table. The CMS gives the First Light team a UI to create, edit, and publish that content without touching the database directly.

---

## 2. Users

All users are **Content Admins** with full read/write access to species, moves, and art.

Auth: email/password via Supabase Auth. No public-facing sign-up.

---

## 3. Core Entities

### 3.1 Bird Species

One record per species (matches `bird_species.species_name` PK used in the app).

| Field | Type | Notes |
|---|---|---|
| Species name | text | Primary key, display name shown on card |
| Scientific name | text | Italic subtitle on card |
| Description | rich text | Shown on bird detail screen |
| Fun facts | list of strings | Shown as rotating trivia on card (up to 5) |
| Migration Speed | int 1–10 | Base stat at level 1; scales with level |
| Endurance | int 1–5 | Base stat at level 1; scales with level |
| Art | image upload (per level) | SVG preferred; one upload per level (1–5) |
| Status | enum | New / Needs Review / Draft / Published |

### 3.2 User Reports

When a First Light user taps "Report an issue" on a bird card in the app, a report is created and the species status is escalated to Needs Review if it isn't already.

| Field | Type | Notes |
|---|---|---|
| Species name | text | FK to bird_species |
| Reported by | user id | The First Light user who filed the report |
| Message | text | Free-text from the user — suggestion, correction, or bug report |
| Created at | timestamp | |
| Resolved | bool | Team marks resolved after addressing |

Reports are visible in the CMS species editor as a feed of comments. Multiple reports can exist per species. The team can mark individual reports resolved without changing the species status, or mark all resolved and move the species back to Draft/Published.

### 3.3 Moves

Each species can have up to 3 moves. Moves are authored per species (not shared globally — the same "Dive Bomb" might have different values on a Peregrine vs. a Tern).

| Field | Type | Notes |
|---|---|---|
| Move name | text | Shown on card |
| Category | enum | Offense / Defense / Support |
| Description | text | What the move does in plain language |
| Effect type | enum | Damage / Speed Boost / Endurance Drain / Draw / Shield / etc. |
| Effect value | int | Numeric magnitude of effect |
| Unlock level | int | Which level (1, 2, or 3) this move becomes available |

### 3.4 Level Progression

Each species has 5 levels. The CMS authors what changes at each level:

| Level | What changes |
|---|---|
| 1 | Base stats (Migration Speed, Endurance), Move 1 available, Level 1 art |
| 2 | Stat increases, Level 2 art unlocks |
| 3 | Move 2 unlocks, Stat increases, Level 3 art unlocks |
| 4 | Stat increases, Level 4 art unlocks |
| 5 | Move 3 unlocks, Max stats, Level 5 art unlocks |

The CMS stores base stat values at level 1 and the delta (increase per level) so the app can compute each level's stats. Art is uploaded once per level — the appropriate image is shown automatically as the player levels up the card.

## 4. Key Screens

### 4.1 Species List

- Searchable, filterable table (filter by status, missing art, missing moves)
- **Needs Review items are sorted first**, followed by New — together these form the team's work queue
- Status badges: `Needs Review` (red) / `New` (amber) / `Draft` (grey) / `Published` (green)
- Quick indicators: art coverage (e.g., 3/5 levels have art), move count, stat completeness
- Bulk actions: publish selected, mark selected as reviewed (moves New/Needs Review → Draft)
- Any species can be manually flagged to Needs Review via a flag icon in the list or the editor
- "New species" button

**Status definitions:**

| Status | Meaning |
|---|---|
| **New** | Auto-generated by AI when a new bird was first logged by a user. Content exists but has not been reviewed by the First Light team. |
| **Needs Review** | Flagged for closer inspection. Triggered by: (1) a First Light user reporting an issue with a bird card from within the app, (2) the system detecting a potential duplicate or unrecognized bird name, or (3) a team member manually flagging it. |
| **Draft** | Being actively edited by the team. Not yet live. |
| **Published** | Reviewed, approved, and live in the app. |

### 4.2 Species Editor

Full-page form. Two-column layout on desktop: fields on left, **live card preview** on right.

Sections:
- **Identity** — name, scientific name, status (with a "Mark as Reviewed" button when status is New or Needs Review; flag button always visible to manually escalate to Needs Review)
- **Stats** — base value + per-level delta for Migration Speed and Endurance, with reference ranges shown (e.g., "Robin Lv1: 3 | Peregrine Lv1: 8")
- **Lore** — description + facts list (add/remove/reorder facts)
- **Art** — 5 upload slots (one per level); drag-and-drop; renders in card preview for the selected level
- **Moves** — 3 move slots locked to unlock levels (1, 3, 5); each collapsible with all move fields
- **Reports** — visible only when status is Needs Review; shows all user-submitted reports as a comment feed with timestamp and username; each report has a "Resolve" button

Auto-save draft. Explicit "Publish" button that makes changes live in the app.

### 4.3 Card Preview

Always-visible panel showing exactly how the card will render in the mobile app. Includes a level selector (1–5) so editors can preview how the card looks and what moves/stats/art are active at each level.

### 4.4 Stats Dashboard

Three tabs: **Birds**, **Product**, and **Suspicious Activity**.

**Birds tab** — game balance view:
- Distribution charts for Migration Speed and Endurance across all published species
- Outlier flagging — highlights species whose stats fall outside the normal range (likely over or underpowered)
- Breakdown by bird family/order so the team can spot imbalances within a group (e.g., all raptors skewing too high in speed)

**Product tab** — user activity view:
- Total users and new signups over time
- Daily/weekly catches logged
- Most-caught species (what birds are players actually seeing in the wild)
- Rarest sightings logged (high sighting_rarity catches)
- Lifer distribution — histogram of how many lifers players have (are people building deep aviaries?)
- Card level distribution — are players leveling up cards or stalling at level 1?
- Feed engagement — pecks and scribbles over time
- Report volume — how many user reports are being filed and how quickly they get resolved

**Suspicious Activity tab** — users are surfaced here for review if they trigger any of the following signals:
- Multiple catches of the same species on the same day (catch-once-per-day rule violated)
- XP totals inconsistent with their catch log history
- Same screenshot hash submitted multiple times across accounts

Flagged users are listed with the specific signal that triggered the flag and their catch history.

## 5. Navigation

```
Sidebar:
  Needs Review     ← flagged by team or system (duplicates, unrecognized birds)
  New              ← AI-generated, not yet touched by the team
  Species          ← full content list
  Moves Library    ← browse/search all moves across species
  Drafts           ← manually edited, unpublished
  Stats            ← bird balance, product metrics, suspicious activity
  Settings         ← user management, Supabase connection
```

## 6. Tech Notes (for dev handoff, not designer)

- Reads/writes directly to Supabase `bird_species`, `species_moves`, and `species_reports` tables
- Art uploads to Supabase Storage bucket `species-art`, keyed by `{species_slug}/level_{n}.{ext}`
- Auth: the CMS signs in with Supabase Auth (anon key + session) like any client. Because app players authenticate against the same project, write access is gated by RLS on membership in the `cms_admins` table (checked via the `is_cms_admin()` security-definer function) — no service role key in the browser.
- Filing a report escalates the species to Needs Review via a database trigger
- Stack: Next.js + Supabase JS client; data access centralized in `cms/lib/api.ts`
